How It Works

Imagine you are building a set of buttons whose background colors come from a design token, a CMS field, or a user preference. Instead of needing paired variables or complex JavaScript to calculate readable text colors, the contrast-color() function hands that job directly to the browser. 

.card {
    --bg: var(--brand-color);
    background-color: var(--bg);
    color: contrast-color(var(--bg));
}

Under the hood, the browser calculates the WCAG 2 contrast ratio of both white and black against your input color. Whichever one produces the higher ratio wins. It accepts any valid color format: 

color: contrast-color(darkblue);
color: contrast-color(#ff6b35);
color: contrast-color(oklch(0.6 0.15 250));

Name Change & History

Earlier drafts of the specification used the name color-contrast() with a different, more complex syntax that accepted a list of multiple candidate colors and a vs keyword. That version was never shipped to stable browsers.

The current function is officially named contrast-color(), takes a single argument, and returns only black or white. The fuller variant with custom candidate lists remains delayed in CSS Color Level 6 specifications.

Why It Matters for Accessibility

The Web Content Accessibility Guidelines (WCAG) require a minimum contrast ratio of 4.5:1 between normal text and its background. Maintaining these ratios across hover states, disabled states, and dark mode variants where background colors shift dynamically is notoriously error-prone.

contrast-color() ensures that the browser will pick whichever of black or white has more contrast with your background. For backgrounds that are clearly light or clearly dark, this produces excellent results.

The Mid-Tone Problem

Consider a medium blue like #317CFF. When run through the WCAG 2 formula, black mathematically wins the contrast ratio calculation over white. However, when you actually look at black text on that blue background, it feels much harder to read than white text.

The WCAG 2 formula calculates luminance contrast mathematically without fully accounting for how human eyes perceive lightness across different color hues. A newer algorithm called APCA (Accessible Perceptual Contrast Algorithm) is being developed for WCAG 3 to address exactly this problem.

Do Not Assume Compliance

contrast-color() is a convenience tool, not a compliance guarantee. It picks the better of two options, but "better" does not always mean "good enough."

For mid-tone backgrounds (roughly 40% to 60% lightness in oklch), you should still test the result manually and consider overriding it with a known accessible pairing when the output looks wrong. Always verify ratios using tools like WebAIM's Contrast Checker.

Practical Implementation Patterns

Dynamic Hover States: When you lighten or darken a background on hover, the text color may need to flip. The function handles this automatically as the token crosses thresholds:

.card {
    --bg: oklch(0.6 0.15 250);
    background: var(--bg);
    color: contrast-color(var(--bg));
}
.card:hover {
    --bg: color-mix(in oklch, oklch(0.6 0.15 250) 40%, white);
}

Feature Detection Safe Fallback: Wrapping code in an @supports block ensures older browsers cleanly fall back to a safe default text color statement:

.badge {
    background-color: var(--badge-bg);
    color: #ffffff; /* fallback */
}
@supports (color: contrast-color(red)) {
    .badge { color: contrast-color(var(--badge-bg)); }
}

Limitations to Be Honest About

  • Only Returns Black or White: You cannot configure a custom branded navy or soft cream color output target.
  • No Font Size/Weight Awareness: It always targets standard 4.5:1 text scales, failing to adapt to lower 3:1 allowances for large or bold titles.
  • Transparent Background Gotcha: It calculates color outputs strictly based on computed token properties. If an element has background: transparent, the function evaluates against invisible black instead of the parents' visible background.
  • No Image/Gradient Evaluation: The function cannot parse dynamic pixel backgrounds or evaluate shifting multi-stop gradients behind text content.
