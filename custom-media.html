The Problem: Repeated Magic Numbers

Media queries are one of the most repeated pieces of CSS in any project. A breakpoint like (min-width: 48em) might appear dozens of times across different component stylesheets: 

/* layout/primary.css */
@media (min-width: 48em) { .site-grid { grid-template-columns: 1fr 3fr; } }

/* components/card.css */
@media (min-width: 48em) { .cards { grid-template-columns: repeat(2, 1fr); } }

The value 48em is a magic number. Custom properties (var()) cannot help here because CSS does not allow variables inside media query conditions. That is the critical gap @custom-media fills.

The Definition & Usage Syntax

A @custom-media rule defines a named alias for a media query. The name must start with two hyphens, just like a custom token property: 

@custom-media --medium (min-width: 48em);
@custom-media --large (min-width: 64em);

/* Referencing the name inside standard rules */
@media (--medium) {
    .cards { grid-template-columns: repeat(2, 1fr); }
}

If design requirements change, you edit one single line, rebuild your site, and the update instantly propagates everywhere.

Where Definitions Belong

In a modern layered CSS architecture, @custom-media configurations belong strictly inside your design token directory (e.g., css/tokens/breakpoints.css). 

/* tokens/breakpoints.css */
@custom-media --small (min-width: 30em);
@custom-media --medium (min-width: 48em);
@custom-media --dark (prefers-color-scheme: dark);

This ensures they sit alongside your color and typography scales so anyone looking for a design token value knows exactly where to find it.

Making It Work in Modern Browsers

As of right now, no browser ships @custom-media in a stable release. Firefox holds it behind a developer flag, while Chrome and Safari have not implemented it natively yet.

To use it today, developers bridge the gap with a polyfill at build time. The build tool reads your modern, clean syntax, replaces it with traditional vanilla CSS media queries, and outputs a compiled file that legacy browsers handle perfectly.

Build Tool Integrations

If using PostCSS: Install the custom plugin tool and add it directly into your postcss.config.js file right after your imports map:

// npm install --save-dev postcss-custom-media
module.exports = {
    plugins: [
        require('postcss-import'),
        require('postcss-custom-media'),
        require('autoprefixer')
    ]
};

If using Lightning CSS: No extra installations are required. Lightning CSS natively parses and flattens @custom-media strings down on compilation out of the box.

Beyond Viewport Breakpoints

Any valid media parameter can be tokenized into an alias. This is incredibly helpful for complex accessibility and interaction queries that are easily mistyped from memory: 

/* Accessibility */
@custom-media --motion (prefers-reduced-motion: no-preference);

/* Input Types */
@custom-media --touch (pointer: coarse);
@custom-media --mouse (hover: hover) and (pointer: fine);

Compilation: Before and After Polyfill Translation

Source (What You Write)

@custom-media --medium (min-width: 48em);

@media (--medium) {
    .cards { grid-template-columns: repeat(2, 1fr); }
}

Output (What the Browser Receives)

@media (min-width: 48em) {
    .cards { grid-template-columns: repeat(2, 1fr); }
}
